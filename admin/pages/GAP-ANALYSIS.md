## 一、需求回顾

用户（CEO Agent）提出的原始需求：
- 东南亚算力机房拜访时记录机房信息（城市、电力、制冷、价格等）
- Agent 辅助网上收集信息，形成填写表
- 工作人员手机端小程序快速填写并上传
- 后台汇总成大表，可导出 Excel
- 微信授权登录 + 权限管理 + 分享导出

本次新增需求（v2.0）：
1. 记录每条机房由哪位同事采集，并可修改
2. 记录每次修改是谁改的、什么时候改的
3. 不同权限的同事可修改不同字段
4. 记录是否已现场考察、谁考察的、未考察则安排考察时间
5. 记录机房预留给哪些客户（可多个）
6. 可按客户筛选机房列表
7. 强大的组合筛选功能

## 三、建议新增（当前设计中未覆盖）

### 1. 按客户维度导出
**现状**：可按城市/Tier筛选导出，但不支持按客户导出
**建议**：新增"客户维度导出"，汇总某客户感兴趣的所有机房，生成客户专属对比表

### 2. Agent 辅助信息抓取（核心亮点）
**现状**：未实现
**建议**：
- 工作人员输入机房名称后，点击"AI 辅助"
- Agent 自动抓取机房官网、Wikipedia、Google Maps 等公开信息
- 预填城市、地址、等级、成立年份、网站等字段
- 工作人员只需核实 + 补充现场信息

### 3. 消息提醒 & 考察通知
**现状**：无通知机制
**建议**：
- 考察日期前 3 天推送提醒
- 新记录创建后通知管理员审核
- 客户跟进状态变更时通知对应客户经理

### 4. 批量操作（管理员）
**现状**：单条编辑
**建议**：
- 批量修改状态（选择多条 → 改为"已放弃"）
- 批量分配客户经理
- 批量设置目标客户

### 5. 数据导入（批量）
**现状**：仅支持单条记录导入
**建议**：
- 支持批量导入已有的 Excel 数据
- 自动去重匹配（按机房名称+城市）
- 导入预览 + 冲突处理

## 四、技术架构建议

### 数据库核心表结构

```
users               — 用户表
idc_records         — 机房记录主表
idc_edit_history    — 修改历史表
idc_customers       — 客户表（预留客户关联）
idc_visits          — 考察记录表
```

### API 接口清单（建议）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/idc` | 新建机房记录 |
| GET | `/api/idc` | 列表（支持组合筛选分页） |
| GET | `/api/idc/:id` | 详情（含修改历史） |

| DELETE | `/api/idc/:id` | 删除（仅管理员） |
| GET | `/api/idc/export` | 导出 Excel |
| POST | `/api/idc/:id/visit` | 新增考察记录 |
| GET | `/api/idc/filter-presets` | 获取预设筛选列表 |
| POST | `/api/idc/filter-presets` | 保存预设筛选 |

## 六、设计原则总结

本产品设计遵循以下原则，确保开发不走偏：

1. **外勤优先**：手机端操作 ≤ 2 分钟完成基本信息录入
2. **数据可信**：所有修改留痕，谁改了什么、什么时候改，不可篡改
3. **权限清晰**：字段级权限，不同角色看得到、改得了的部分不同
4. **考察闭环**：从"未考察"到"已考察"，全流程可追踪
5. **客户导向**：机房不只是机房，还关联目标客户和跟进状态
6. **筛选高效**：支持多维度组合筛选，5 秒内找到目标机房
